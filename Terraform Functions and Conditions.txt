Terraform functions work interactively in the console.
Here’s what’s happening:

🧩 Key Functions Demonstrated
file() — reads the contents of a file. In this case, it loads /root/terraform-projects/main.tf, which defines an AWS instance resource.

length() — returns the number of elements in a list or map. Here, length(var.region) outputs 3, meaning the variable region contains three items.

toset() — converts a list to a set, removing duplicates. The output shows two unique regions: ca-central-1 and us-east-1.

Transforming data types (lists → sets).

Inspecting configuration files directly from the console.

That image is a great illustration of how Terraform workspaces can dynamically control configuration values — especially AMIs — across different environments or projects.

Here’s what’s happening in that setup:

🧩 Concept Overview
Terraform workspaces let you maintain multiple independent state files within the same configuration.
Each workspace acts like a separate environment (e.g., ProjectA, ProjectB, dev, prod).

You can check your current workspace with:

bash
terraform workspace show
or in the console:

bash
terraform.workspace
⚙️ How the Example Works
variables.tf
hcl
variable "ami" {
  type = map(string)
  default = {
    "ProjectA" = "ami-0edab43b6fa892279"
    "ProjectB" = "ami-0c2f25c1f66a1ff4d"
  }
}
main.tf
hcl
resource "aws_instance" "projectA" {
  ami           = lookup(var.ami, terraform.workspace)
  instance_type = var.instance_type

  tags = {
    Name = terraform.workspace
  }
}

bash

✅ When you switch to workspace ProjectA, Terraform automatically picks the AMI for ProjectA.
✅ When you switch to ProjectB, it uses the AMI for ProjectB.

🧠 Why this is powerful
You can reuse the same configuration for multiple environments.

Each workspace keeps its own state file, so resources don’t conflict.

The lookup() function dynamically selects values based on the active workspace.
