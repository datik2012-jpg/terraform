//////////////////////////Terraform State Commands //////////

Incorrect
bash
vi terraform.tfstate
Editing the state file manually is not recommended.
Terraform’s state file (terraform.tfstate) is a machine‑managed record of all resources.
Manual edits can corrupt it, break resource tracking, or cause drift between real infrastructure and Terraform’s view.



Correct
bash
terraform state show aws_s3_bucket.finance
This is the proper way to inspect a resource’s state.
It safely queries Terraform’s internal state and displays the attributes of the resource aws_s3_bucket.finance without modifying anything.


Command	Purpose
terraform state list	Lists all resources tracked in state
terraform state show	Displays details of one resource
terraform state rm	Removes a resource from state safely
terraform state mv	Moves a resource between modules or states


terraform state rm <resource>
does NOT destroy the resource in the cloud.  
It only removes Terraform’s tracking of that resource from the state file.

What actually happens
Action	Effect on Cloud	Effect on Terraform
terraform destroy	Deletes the resource from AWS/Azure/GCP	Removes it from state
terraform state rm	Leaves the resource running	Removes it from state only

Step 1 — Original resource definition
In main.tf:

hcl
resource "aws_dynamodb_table" "state-locking" {
  name         = "state-locking"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"
  attribute {
    name = "LockID"
    type = "S"
  }
}
This defines a DynamoDB table named state-locking used for Terraform state locking.

🧩 Step 2 — State file mismatch
In terraform.tfstate, Terraform currently tracks:

json
{
  "type": "aws_dynamodb_table",
  "name": "state-locking-db"
}
So the state file expects the resource name state-locking-db, not state-locking.

This mismatch happens when you rename a resource in your .tf file but haven’t updated the state yet.

🧩 Step 3 — Fixing the mismatch
The command:

bash
terraform state mv aws_dynamodb_table.state-locking aws_dynamodb_table.state-locking-db
moves the resource reference inside Terraform’s state from state-locking → state-locking-db.

Terraform confirms:

Code
Move "aws_dynamodb_table.state-locking" to "aws_dynamodb_table.state-locking-db"
Successfully moved 1 object(s).
Now your configuration and state file are synchronized.

🧠 Why this matters
Prevents Terraform from thinking the resource was deleted and recreating it.

Keeps your infrastructure stable while renaming resources.

Avoids manual edits to the .tfstate file (which can corrupt it).

📌 Summary table
Command	Purpose	Safe?
terraform state mv	Renames or moves a resource in state	✅
terraform state rm	Removes resource from state (does not destroy)	⚠️ Use carefully
terraform import	Adds existing resource back into state


So instead of renaming, it will destroy the existing resource and create a new one — which is risky for production infrastructure.

When you run terraform apply, Terraform sees:

state-locking → missing from config → destroy it

state-locking-db → new in config → create it

Result: two operations — destroy + create — instead of a simple rename.

That’s why you use:

bash
terraform state mv aws_dynamodb_table.state-locking aws_dynamodb_table.state-locking-db
This tells Terraform:

“It’s the same resource, just a new name.”

No destruction, no downtime, no data loss.

Why use this command
✅ Safe when:

You manually deleted the resource in AWS.

You want Terraform to stop managing it.

You plan to re‑import it later with terraform import.

🚫 Risky when:

You still want Terraform to manage the resource — it will recreate it on the next apply.
