////////////////////////Mutable vs Immutable Ifrastructure //////

Mutable vs. Immutable Infrastructure is one of those foundational Terraform concepts that completely changes how you think about deploying systems. Let’s break it down in a way that’s clear, practical, and tied to how Terraform actually behaves.


Mutable infrastructure means you change servers in-place.
You update, patch, reconfigure, or modify an existing resource instead of replacing it.


How it behaves
You SSH into a server and update software

Terraform updates an existing resource instead of recreating it

State stays the same, resource ID stays the same

Pros
Fast updates

No need to recreate everything

Works well for databases or long-lived systems

Cons
Configuration drift

Harder to reproduce environments

“Snowflake servers” — each one becomes unique over time

erraform example (mutable)
hcl
resource "aws_instance" "web" {
  ami           = "ami-123"
  instance_type = "t3.micro"  # Changing this updates in-place
}

Immutable Infrastructure
Immutable infrastructure means you never modify servers.
Instead, you replace them with new ones.

How it behaves
You build a new AMI or container image

Terraform destroys the old resource and creates a new one

No drift, fully reproducible

Pros
Predictable deployments

Easy rollbacks

No drift

Works perfectly with CI/CD, autoscaling, containers

Cons
Slower deployments

More resource churn

Not ideal for stateful systems (databases)


Terraform example (immutable)

resource "aws_instance" "web" {
  ami           = "ami-new-version"  # Changing AMI forces replacement
  instance_type = "t3.micro"
}

Terraform will show:

Code
-/+ destroy and create replacement



Terraform’s philosophy:

“If a change risks breaking the resource, replace it.”

But Terraform can do mutable updates when the provider supports it.

When to Use Which
Use Mutable when:
Databases

Stateful servers

Systems with persistent storage

You need fast, in-place updates


Use Immutable when:
Web servers

Containers

Auto-scaling groups

Blue/green deployments

CI/CD pipelines


Concept	Mutable	Immutable
Terraform behavior	Update in-place	Destroy + recreate
Drift	High risk	No drift
Speed	Fast	Slower
Rollbacks	Hard	Easy
Best for	DBs, stateful	Web apps, stateless

////////////////////////Licycles Rules in Terraform ///////////

Terraform alsways by default delete the resource and deploy new when we run APPLY

Sometimes we do not want the resource to be deleted and want to do som changes on the current resource  - muttable option in this case , we modify current resource

For example we want create the resource before we destroy the old one:

resource "local_file" "pet" {
  filename        = "/root/pets.txt"
  content         = "We love pets!"
  file_permission = "0700"

  lifecycle {
    create_before_destroy = true
  }
}

This tells Terraform:

“When this resource changes, create the new one before destroying the old one.”

Another Example: If you want Terraform to update a resource in place (change its attributes without destroying and recreating it), you can control that behavior using the lifecycle meta‑argument.

lifecycle {
  prevent_destroy = true
}


This tells Terraform:

“Never destroy this resource — even if the configuration changes.”


If you modify attributes that normally trigger replacement, Terraform will throw an error instead of deleting the resource.



resource "local_file" "pet" {
  filename        = "/root/pets.txt"
  content         = "We love pets!"
  file_permission = "0700"

  lifecycle {
    prevent_destroy = true
  }
}

Now, if you change content or file_permission, Terraform will update the file in place.
If you try to delete or recreate it, Terraform will stop you with an error.


Other lifecycle options
Lifecycle rule	Behavior
create_before_destroy	Creates new resource before destroying old one (avoids downtime)
prevent_destroy	Blocks destruction entirely
ignore_changes	Ignores specific attribute changes (useful for external updates)

Example:

lifecycle {
  ignore_changes = [file_permission]
}

erraform will skip updates to that attribute even if it changes outside Terraform.

🧭 Summary
To change a resource without destroying it, use one of these:

prevent_destroy = true → never delete

ignore_changes = [...] → skip updates for certain attributes

create_before_destroy = true → replace safely without downtime

///////////////////////////////Data sources /////////////

That snippet defines a Terraform data source, not a resource — and that distinction is key.

Here’s what it means:

data "local_file" "dog" {
  filename = "/root/dog.txt"
}


data block
A data source lets Terraform read existing information rather than create something new.
In this case, it reads the contents of a local file that already exists on your system.

🔹 local_file data source
This comes from the local provider. It’s used to import file content into Terraform so you can reference it elsewhere — for example, to use the file’s text in another resource.

🔹 filename
Specifies the path to the file Terraform should read (/root/dog.txt).



What happens when you run Terraform
When you run terraform apply or terraform plan, Terraform will:

Look for /root/dog.txt

Read its contents

Make that content available as an output variable

You can access it like this:

hcl
output "dog_content" {
  value = data.local_file.dog.content
}


Summary
Element	Purpose
data	Reads existing info
local_file	Reads local file content
filename	Path to file
data.local_file.dog.content	Returns file text

//////////////////////////////for_each/////////////////

Perfect observation, Dani — this Terraform setup is the for_each version of the dynamic file creation pattern you explored earlier. Let’s break down exactly how it works and why it’s often preferred over count.


resource "local_file" "pet" {
  filename = each.value
  for_each = toset(var.filename)
}

for_each
This tells Terraform to iterate over each element in the set created from var.filename.
Each iteration creates a separate local_file resource.

🔹 each.value
Represents the current item in the loop — in this case, the filename string.

So Terraform will create:

/root/pets.txt

/root/dogs.txt

/root/cats.txt

Each file becomes its own resource, referenced as:

hcl
local_file.pet["/root/pets.txt"]
local_file.pet["/root/dogs.txt"]
local_file.pet["/root/cats.txt"]
🧩 variables.tf
hcl
variable "filename" {
  type = list(string)
  default = [
    "/root/pets.txt",
    "/root/dogs.txt",
    "/root/cats.txt"
  ]
}
This defines a list of strings — each one is a file path.
Terraform converts it to a set with toset(var.filename) so for_each can iterate over unique values.

🧠 Why use for_each instead of count
Feature	count	for_each
Indexing	Numeric (count.index)	Keyed (each.key)
Uniqueness	Allows duplicates	Requires unique keys
Resource references	resource[0], resource[1]	resource["key"]
Best for	Simple lists	Named or unique items
